④ 商品開発AIエージェント（新設・別系統）

データ基盤＝統合台帳_v2（13タブ）
成分マスタ・販売実績・候補プール・シグナル・競合・マーケ仮説・ボツ案などを1つのスプレッドシートに集約。 販売実績マスタにはShopify実売を反映済み（直近月商・売れ筋順位入り）。
0-1 設定0-2 情報収集設定（会社が記入）1 成分マスタ 4 候補プール（提案）5 市場シグナルログ6 競合商品ウォッチ 7 マーケ仮説KPI9 販売実績マスタ（実売反映）10 ボツ案 …ほか計13タブ
売れ筋実績（直近30日）: VOLCANO 2.0 ¥1,008,000(1位) / BLACK HOLE ¥792,000(2位) / THA REAL SNOOP OG ¥720,000(3位) / XTC ¥630,000(4位) … ＝候補提案の根拠データ。
情報収集は「AIが推測しない」設計
会社（社長/マーケ/開発2名）が統合台帳の新タブ 「0-2_情報収集設定」 に 監視する競合・Xアカウント/界隈・検索キーワード を記入 → AIはそこに書かれた対象だけを毎日巡回（cron A・X API）。 記入が増えるほど提案精度が上がる。空欄時は規制監視のみの最小フォールバックで動く。

データの流れ（生成 → 関所 → 書込 → 逆フィードバック）

① 情報収集（自動）
0-2設定の競合X/界隈/KWを巡回 → 5_シグナル・6_競合へ自動追記（観測ログ）

② 候補生成（週次・月曜）
シグナル＋競合＋9_販売実績＋1_成分マスタを読み、新商品候補を2〜3件。15項目必須＋過去ボツ/既存重複の自動チェック＋市場フィット採点
↓
③ 関所＝社長承認
候補（4）・マーケ仮説（7）は台帳に書かずまずSlackで提示。承認後のみ台帳へ追記
→
④ 書込・商品化
承認分を 4_候補 / 7_仮説 へ記入 → 制作物テンプレ（商品情報/Shopify説明/X/価格物流 等）→ 発売
↻ 逆フィードバック
⑤ 発売 → 売上 → 成分マスタへ逆FB
発売後の実売（9_販売実績の直近月商・売れ筋順位）を 1_成分マスタ の評価へ還元 → 次の候補生成の精度が上がる学習ループ
←
継続観測
シグナル/競合は毎日更新され、夜レポートで重要度A（規制・大型競合・需要急増）を先頭表示

⑤ 承認の住み分け（混同しないために）

🟢 秘書AIの承認 ＝ #ai-承認待ち
  • 夜レポート・タスク・入金確認・CS下書き・X DM 等、日常業務の判断
  • 宮さんが OK / NG / 修正 をスレ返信 → 平日9/12/15/18/21時の承認監視が拾って実行
  • 窓口は #ai-承認待ち（C0ATPFSLM4M）／日次まとめは #ai-日報
🟣 商品開発の承認 ＝ 台帳・開発専用
  • 新商品候補（4）・マーケ仮説（7）の採否は開発専用の承認動線で扱う
  • 秘書の #ai-承認待ち には入れない（業務承認と混ざらないよう分離）
  • 承認されたものだけが 台帳シート4 提案中 へ書き込まれる＝関所
理由: 商品開発は判断の質・回数が日常承認と異なるため、専用の場で集中して判断できるよう動線を分けている。
